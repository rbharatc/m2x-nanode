1 # Nanode M2X API Client #
2 
3 The AT&T M2X API provides all the needed operations to connect your device to AT&T's [M2X](http://m2x.att.com) service. This client provides an easy to use interface for [Nanode](http://www.nanode.eu) microcontroller based devices. This client library supports writing data from a Nanode devices to M2X but does not support reading data back out from the M2X service.
4 
5 ## Getting Started ##
6 
7 1. Signup for an [M2X Account](https://m2x.att.com/signup).
8 2. Obtain your _Master Key_ from the Master Keys tab of your [Account Settings](https://m2x.att.com/account) screen.
9 3. Create your first [Device](https://m2x.att.com/devices) and copy its _Device ID_.
10 4. Review the [M2X API Documentation](https://m2x.att.com/developer/documentation/overview).
11 
12 Please consult the [M2X glossary](https://m2x.att.com/developer/documentation/glossary) if you have questions about any M2X specific terms.

15 ## How to Use the Library ##
16 
17 1. Clone [ethercard](https://github.com/xxuejie/ethercard) library:
18 
19  $ git clone https://github.com/xxuejie/ethercard
20 
21 2. Open the Arduino IDE, click `Sketch`->`Import Library...`->`Add Library...`, navigate and select the ethercard repo folder, click `OK` to import the libray into Arduino.
22 
23 3. Clone [m2x-nanode](https://github.com/attm2x/m2x-nanode) library:
24 
25  $ git clone https://github.com/attm2x/m2x-nanode
26 
27 4. Use the process outlined in Step #2 above to import `M2XNanodeClient` in this repo. Note that while in Step #2, we are selecting the repo folder `ethercard`, in this step we are selecting a sub-directory of the repo.
28 
29 5. Open one of the Nanode examples in `File`->`Examples`->`M2XNanodeClient`.
30 
31 6. Change `Device ID`, `Stream Name` and `M2X Key` fields to contain correct values (from your M2X account).
32 
33 7. Set the board type to Arduino UNO via `Tools` -> `Board Type` -> `Arduino UNO`.
34 
35 8. Now you can run the examples!
36 
37 **NOTE**: For simplicity, all of provided examples use fixed values as inputs. If you want to use values obtained from real sensors as inputs, please refer to [this tutorial](http://shop.wickeddevice.com/2013/10/16/nanode-gatewayremote-sensing-tutorial-1/) for the value gathering functionality.

39 ## Client API ##
40 
41 Due to the memory limitations of Nanode boards, most of the APIs provided work in a callback-based approach:
42 
43 1. You define one or more function(s) to provide the data needed.
44 
45 2. You pass the function(s) as parameter(s) to API calls.
46 
47 3. The library will invoke provided function(s) as callbacks to fill in necessary data.
48 
49 For more details on how this works, please refer to the examples.
50 
51 ### UpdateStreamValue API ###
52 
53 The UpdateStreamValue API is the simplest approach to pushing data into M2X. The API interface is as follows:
54 
55 ```
56 typedef void (*put_data_fill_callback)(Print* print);
57 int updateStreamValue(const char* device_id, const char* stream_name,
58  put_data_fill_callback cb);
59 ```
60 
61 A sample callback function will look like the following:
62 
63 ```
64 static int val = 11;
65 void fill_data_cb(Print* print) {
66  print->print(val);
67 }
68 ```
69 
70 If you want to pass strings as values, you need to include double-quotes:
71 
72 ```
73 void fill_data_cb(Print* print) {
74  print->print("\"11\"");
75 }
76 ```
77 
78 ### PostStreamValues API ###
79 
80 PostStreamValues API has the following differences from UpdateStreamValue API:
81 
82 1. You can push multiple values in one request;
83 2. For each value, you also need to provide an ISO8601-formatted timestamp.
84 
85 The calling interface for this API is as follows:
86 
87 ```
88 typedef void (*post_data_fill_callback)(Print* print, int index);
89 int postStreamValues(const char* device_id, const char* stream_name, int value_number,
90  post_data_fill_callback timestamp_cb,
91  post_data_fill_callback data_cb);
92 ```
93 
94 Notice here that we use two callback functions: one for timestamp, and one for data. For each callback function, the client library will invoke it `value_number` times, each time for a different value to be pushed.
95 
96 Notice for timestamp and string-typed values, double quotes are needed:
97 
98 ```
99 static int seconds = 10;
100 void fill_timestamp_cb(Print* print, int index) {
101  print->print("\"2014-07-09T19:");
102  print->print(15 + index);
103  print->print(":");
104  print->print(seconds);
105  print->print(".624Z\"");
106 }
107 ```
108 
109 ### PostDeviceUpdates API ###
110 
111 PostDeviceUpdates API has one more advantage over the PostStreamValues API: it allows pushing multiple values to multiple streams in one request. As a result, the calling interface for this API is more complex:
112 
113 ```
114 typedef int (*post_multiple_stream_fill_callback)(Print* print, int stream_index);
115 typedef void (*post_multiple_data_fill_callback)(Print* print, int value_index, int stream_index);
116 int postDeviceUpdates(const char* device_id, int stream_number,
117  post_multiple_stream_fill_callback stream_cb,
118  post_multiple_data_fill_callback timestamp_cb,
119  post_multiple_data_fill_callback data_cb);
120 ```
121 
122 First, `stream_number` denotes how many streams we are pushing to. For each stream, `stream_cb` will be called with the corresponding index, this callback function has 2 effects:
123 
124 1. It prints the stream name into `print`(double quotes needed);
125 2. It returns the number of values we want to push for this stream.
126 
127 Suppose during the `i`th time we call `stream_cb`, `j` is returned as a result. Then we will call `timestamp_cb` callback function `j` times, with `i` as `stream_index`, and `0` through `j-1` as `value_index`, each time, we are getting the timestamp for the value to push.
128 
129 We are also calling `data_cb` the same number of times with the same input parameter to get the value to push.
130 
131 Notice that for all callback functions, if string values are printed, double quotes are needed.
132 
133 ### Update Location API ###
134 
135 The update Location API can be used to update the location of a device. The calling interface is as follows:
136 
137 ```
138 // Values of data type:
139 // 1 - Latitude
140 // 2 - Longitude
141 // 3 - Name
142 // 4 - Elevation
143 typedef void (*update_location_data_fill_callback)(Print* print, int data_type);
144 int updateLocation(const char* device_id, int has_name, int has_elevation,
145  update_location_data_fill_callback cb);
146 ```
147 
148 In this API request, `name` and `elevation` are optional values. `has_name` and `has_elevation` thus determine if `name` and `elevation` are present respectively: 1 for present, 0 for absent.
149 
150 Depending on the values `has_name` and `has_elevation`, the callback function will be called 2 - 4 times. Each time, a different `data_type` value is used, so we can know which value is requested right now.
151 
152 Like all the requests outlined above, if string-typed values are printed, double quotes are needed as well.
153 
154 ### Delete Values API ###
155 
156 Delete Values API can be used to delete values within a specific time range. The calling API is as follows:
157 
158 ```
159 // Values for timestamp type:
160 // 1 - Start Time
161 // 2 - End Time
162 typedef void (*delete_values_timestamp_fill_callback)(Print* print, int type);
163 int deleteValues(const char* device_id, const char* stream_name,
164  delete_values_timestamp_fill_callback timestamp_cb);
165 
166 ```
167 
168 Following is an example of the callback function:
169 
170 ```
171 void fill_timestamp_cb(Print* print, int type) {
172  if (type == 1) {
173  print->print("\"2014-07-01T00:00:00.000Z\"");
174  } else {
175  print->print("\"2014-07-30T00:00:00.000Z\"");
176  }
177 }
178 ```

180 ## Known Issues ##
181 
182 * In our tests with Nanode based devices, we found that there is a small chance that an API request may timeout. This occurs inside the ethercard library: our internal callback functions are not called at all. We suspect that this may be related to the way TCP/IP is implemented in the library, or our way of using the library (we might accidently set the wrong parameter for some option).
183 
184 * Unlike the Arduino client library, fetching values is not supported in the Nanode library. This is because [ethercard](https://github.com/xxuejie/ethercard) does not support packet streaming: when a packet arrives, ethercard will read all the data into memory before handling over control to our code. Considering the fact that a Nanode only has 2kb memory, and that a simple M2X List Value API response contains 4k-5k data, we can never process this request on Nanode successfully.
